## Методология и применение

**User Story Map (Jeff Patton):** карта путешествия пользователя во времени (слева→направо) + приоритизация по важности (сверху→вниз). Три слоя:

- **Backbone (activities):** крупные фазы пути клиента. Меняются медленно.
- **Tasks:** что клиент **делает** на каждой activity. Выражены глаголом, **не наши фичи**.
- **Stories:** что **мы строим**, чтобы клиент справился с task'ом. Приоритезируются по `$ impact` + релизу (R1/R2/R3).

**Принцип Алана (инсайт #2 из refounding):** каждая story имеет `$ impact` (High/Med/Low). High = прямой драйвер retention/acquisition/churn. Топ-5 R1 stories обязаны быть про деньги.

**Принцип одного actor'а для R1:** в сегменте 15-40 номеров Владелец-собственник физически обрабатывает WhatsApp сам в 80% случаев. Двухрядный backbone (Владелец + Менеджер) — overengineering для R1. Вторая роль вводится в R2 вместе с `Operator`-моделью и двусторонним каналом.

## Backbone (Layer 1) — 7 activities

```
┌──────────┐  ┌───────────┐  ┌────────┐  ┌──────┐  ┌──────────┐  ┌──────┐  ┌────────┐
│ 1. Hear  │→ │ 2. Eval.  │→ │ 3. Try │→ │ 4. Buy│→│ 5. Activ. │→ │ 6. Run│→ │ 7. Renew│
└──────────┘  └───────────┘  └────────┘  └──────┘  └──────────┘  └──────┘  └────────┘
```

**Почему именно так (выходы party-mode):**

- **Trial выделен отдельно** (Mary): между Evaluate и Buy есть фаза «Назира на встрече + владелец пробует demo-номер 1-2 дня». Это активность с своими stories (demo-бот, preview-chat, share-ссылка), а не шаг внутри Evaluate.
- **Activate выделен отдельно** (Victor): классический SaaS теряет 30-40% клиентов между Buy и первым успешным диалогом. Activate = первые 7-14 дней после оплаты, инженеримся вокруг этого окна (гайды, анти-бан чеклист, «первое сообщение доставлено» checkmark).
- **Run двухрежимный R1/R2** (Winston): до июня 2026 Exely read-only → R1 flow = «бот ловит лид, менеджер оформляет». С июня Exely открывает API → R2 flow = «бот сам закрывает сделку». Разные наборы stories под одной activity.

## Tasks layer (Layer 2) — что владелец делает

| Activity | Tasks (действия владельца) |
|---|---|
| **1. Hear** | Получает реферал от Назиры · Гуглит «AI бот для отеля Бишкек» · Слышит от знакомого отельера |
| **2. Evaluate** | Сравнивает с NURAI / делает сам · Смотрит цену · Читает Ton Azure кейс · Спрашивает про Exely/Altegio |
| **3. Trial** | Смотрит демо на встрече с Назирой · Запускает preview-chat на своих данных · Показывает партнёру/бухгалтеру |
| **4. Buy** | Получает счёт · Оплачивает · Подписывает акт/договор |
| **5. Activate** | Заполняет визард · Покупает отдельный номер + 7-14 дней прогрева · Подключает в Wappi · Подключает TG-бот · Тестит первое сообщение · Настраивает менеджеров |
| **6. Run** | Получает заявки · Передаёт менеджеру · Смотрит метрики · Правит промпт (staging→promote) · Читает месячный ROI-отчёт |
| **7. Renew** | Видит следующий счёт · Сравнивает ROI · Апгрейдит tier · Рекомендует знакомому |

**Итого: 27 tasks (DoD ≥10 перевыполнен).**

### Три критические task'а (флаги команды)

1. **Evaluate → «Спрашивает про Exely/Altegio»** (Victor). Наш главный differentiator vs NURAI (general-purpose AI). Провал на этом task'е = проигранная сделка до Trial. **⚠️ Shelter исключён из R1** — интеграции нет, только outreach-намерение.
2. **Activate → «Покупает отдельный номер + 7-14 дней прогрева»** (Mary + Wappi-переписка Алана с разрабами 2026-04-21). Здесь отвал 30-40% без человеческой помощи. Wappi работает на сервере — телефон дома, но анти-бан правила жёсткие (не более 5-10 msg/мин, персонализация, прогретая история).
3. **Renew → «Читает месячный ROI-отчёт»** (John). Если владелец не видит «я бы потерял 3 брони = $900 без бота» — churn на 3-й месяц. Это retention-core, не украшательство.

## UX-развилка (Sally) — #20 и #22 в визарде

**Проблема:** Реквизиты оплаты (#20) и роли менеджеров (#22) сейчас вынесены в activity `Activate`, но владелец столкнётся с ними **в визарде** (до Buy).

**Два варианта:**

1. **Required в визарде:** владелец не может завершить визард без реквизитов/ролей. Плюс: данные собраны 100%. Минус: потеря конверсии 10-20% на визарде.
2. **Optional в визарде + WA-дожим:** визард заполняется быстро, недостающие данные Назира запрашивает по WA в первые 48ч после оплаты. Плюс: конверсия. Минус: часть владельцев игнорирует дожим → первый диалог всё равно ломается.

**Дефолт принят:** optional + WA-дожим, с **техническим барьером** — бот не отправляет сообщение клиенту с `[РЕКВИЗИТЫ]` если поле пустое, вместо этого уведомляет владельца «заполните реквизиты, бот не может ответить клиенту».

Это **fail-loud паттерн** — лучше пропустить диалог, чем отправить сломанный ответ.
